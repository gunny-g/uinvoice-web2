# Task 007-1: 핵심 기능 통합 테스트

## 개요

Notion 실데이터 연동 완료 후 Playwright MCP를 사용하여 전체 사용자 플로우,
엣지 케이스, 에러 핸들링을 종합 검증합니다.

## 의존성

Task 007 완료 후 진행

## 수락 기준

- [ ] 목록 → 상세 → PDF 저장 전체 플로우 오류 없이 동작
- [ ] Notion API 매핑 정확성 검증 (금액·날짜·상태 포맷)
- [ ] 모든 엣지 케이스 시나리오 통과
- [ ] 공개 라우트 동작 확인 (인증 없이 접근 가능)

## 테스트 시나리오

### 1. 정상 플로우 (Happy Path)

- [ ] `/invoices` 접속 → 목록 페이지 정상 렌더링
- [ ] 발행일 내림차순 정렬 확인
- [ ] 견적서 카드 클릭 → `/invoices/[id]` 이동
- [ ] 상세 페이지: 헤더 정보 (견적번호, 클라이언트명, 발행일, 유효기간, 상태) 렌더링
- [ ] 상세 페이지: 견적 항목 테이블 (항목명, 수량, 단가, 금액) 렌더링
- [ ] 상세 페이지: 총금액 렌더링
- [ ] "← 목록으로" 링크로 목록 페이지 복귀
- [ ] "PDF 저장" 버튼 클릭 → 인쇄 대화상자 열림

### 2. 데이터 매핑 정확성

- [ ] `총금액` Number 필드 → `formatCurrency` 포맷 일치 여부 (예: `1,500,000원`)
- [ ] `발행일` / `유효기간` Date 필드 → `formatDate` 포맷 일치 여부 (예: `2025.01.15`)
- [ ] `상태` Select 값 → StatusBadge 색상 매핑 (`대기`/`승인`/`거절`)
- [ ] `수량 * 단가` = `금액` Formula 계산 결과 일치

### 3. 에러 핸들링 및 엣지 케이스

- [ ] 존재하지 않는 UUID 형식 ID → 404 not-found 페이지
- [ ] 완전히 임의의 문자열 ID (`/invoices/abc`) → 404 not-found 페이지
- [ ] `항목` Relation이 비어 있는 견적서 → 빈 항목 테이블, 에러 없음
- [ ] `총금액` 필드가 null인 견적서 → "—" 또는 "0원" 표시, 에러 없음
- [ ] `상태` 필드가 null / 미정의 값인 견적서 → 뱃지 없음 또는 기본 표시

### 4. 공개 라우트 확인

- [ ] 세션/쿠키/인증 헤더 없이 `/invoices` 접근 가능
- [ ] 세션/쿠키/인증 헤더 없이 `/invoices/[실제ID]` 접근 가능
- [ ] 브라우저 시크릿 모드에서 동일 동작 확인 (수동)

### 5. ISR 캐싱 동작 (선택)

- [ ] 페이지 소스에서 `Cache-Control` 헤더 확인 (프로덕션 빌드 시)

## 검증 방법

```
Playwright MCP 순서:
1. browser_navigate → http://localhost:3000/invoices
2. browser_snapshot → 목록 렌더링 스냅샷
3. browser_click → 첫 번째 견적서 카드
4. browser_snapshot → 상세 페이지 렌더링 스냅샷
5. browser_navigate → http://localhost:3000/invoices/invalid-id
6. browser_snapshot → 404 페이지 스냅샷
```
